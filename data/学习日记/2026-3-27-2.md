# 📑 企业级制度 RAG 问答系统技术文档

## 1. 项目概述

本系统是一个基于 LangChain 框架开发的 RAG（检索增强生成）原型，旨在解决公司内部规章制度（PDF 格式）的智能检索与问答需求。系统集成了阿里 DashScope 向量化与生成模型，具备完整的增量数据更新与溯源功能。

## 2. 技术架构与选型

系统采用典型的 RAG 架构，并针对生产场景进行了工程化增强：

- **向量化模型**：阿里 text-embedding-v4（支持 2048 维向量，适合复杂语义）。
- **大语言模型**：qwen-turbo / deepseek-v3（通过 DashScope API 调用）。
- **向量数据库**：FAISS（本地轻量化部署，支持 HNSW 索引）。
- **PDF 解析**：PyPDF2（支持按页码提取文本，便于 Step 6 的溯源）。

## 3. 核心功能模块设计

### 3.1 数据清洗与分块 (ETL)

为了保证检索的精度，系统采用了递归字符分割器：

- **Chunk Size**：500 字符（平衡了语义完整性与检索颗粒度）。
- **Chunk Overlap**：50 字符（通过冗余重叠防止核心信息在切分处断裂）。
- **清洗逻辑**：自动剔除 PDF 中的不可见控制字符（如 `\x00-\x1f`）及重复换行，提升 Embedding 质量。

### 3.2 智能增量更新机制 (Incremental Sync)

这是本系统的核心工程亮点。系统不依赖简单的文件名判断，而是使用 MD5 文件指纹：

- **扫描**：遍历 `./source` 目录下的所有 PDF。
- **比对**：提取当前文件 MD5 值，与 FAISS 库中存储的 `file_hash` 元数据对比。
- **决策**：
  - **Hash 匹配**：跳过，不重复调用 API，节省成本。
  - **Hash 不匹配**：删除旧片段，重新 Embedding 并追加到库中。

### 3.3 相似度度量与安全过滤

系统将底层数学距离转换为业务可感知的相似度分数：

- **转换公式**：$Similarity = 1 / (1 + L2\_Distance)$
- **动态防护栏**：设置 `_DEFAULT_SIMILARITY_THRESHOLD = 0.4`。当最高相似度低于 0.4 时，系统判定为"知识库未覆盖"，自动触发拒答机制，防止 LLM 产生幻觉。

## 4. 问答策略 (Prompt Engineering)

为了确保答案的严谨性，系统使用了结构化的 System Prompt：

```plaintext
# ROLE: 严谨的公司行政助手
# CONSTRAINTS:
1. 仅根据【已知制度】回答，不得加入主观臆断。
2. 无法回答时，必须直说"库中暂无相关规定"。
3. 必须在回答末尾标注参考的文件名与页码。
```

## 5. 命令行使用手册

系统支持多命令模式，兼顾"开发调试"与"用户交互"：

| 命令 | 功能描述 | 使用场景 |
|------|----------|----------|
| `python langchain_rag.py` | 进入交互式问答模式 | 终端用户直接对话 |
| `python langchain_rag.py sync` | 强制同步知识库 | 更新了 PDF 文档后运行 |
| `python langchain_rag.py search [query]` | 打印检索到的原始片段 | 调试检索质量、观察分数 |
| `python langchain_rag.py ask [query]` | 单次问答模式 | 脚本化调用或集成到其他系统 |

## 6. 开发者避坑指南 (Troubleshooting)

- **MacOS 环境冲突**：脚本已通过 `os.environ["KMP_DUPLICATE_LIB_OK"] = "TRUE"` 解决了 libomp 重复初始化导致的程序崩溃问题。
- **API 频率限制**：针对 DashScope 偶发的 SSLError 或网络超时，代码内置了 **5 次指数退避重试**逻辑。
- **向量库加载安全**：由于使用了 Pickle 序列化，代码显式启用了 `allow_dangerous_deserialization=True`。

## 7. 总结

本系统通过元数据溯源 (Metadata) 和 哈希校验 (MD5) 两大技术手段，成功将一个简单的 Demo 升级为具备商业参考价值的规章制度问答工具。它解决了 RAG 在实际落地中"不可信（幻觉）"和"不可控（重复开销）"的两大痛点。
