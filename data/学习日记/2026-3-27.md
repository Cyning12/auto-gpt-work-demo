## 二、RAG 核心原理与 Native 流程

### 2.1 检索 + 生成（Retrieval + Generation）

RAG 从外部知识库检索相关信息，作为 **Context** 输入 LLM，以提升答案的事实性与可核对性。  
关联理解：将《三国演义》等文本切片写入 FAISS 属于 **检索侧**；把检索到的 Snippet 拼进 Prompt 再调用模型属于 **生成侧**。

### 2.2 Native RAG 标准流程

1. **数据准备**：Load → Split → Embedding → Vector Store  
2. **检索**：Similarity Search（及后续的阈值、过滤等）  
3. **生成**：Prompt + Context → LLM

**关键参数**：

- **Chunk Size**：控制信息粒度（例如从 3500 调到 **500** 以减轻语义稀释）。  
- **Chunk Overlap**：保证切分边界处语义连贯，减少“拦腰截断”（笔记中采用 **50** 字符量级与递归分割配合）。

### 2.3 常见问题与质量提升

| 问题 | 说明 | 可能手段 |
|------|------|----------|
| **语义偏移** | 向量匹配差时召回不相关片段（如搜“桃园结义”却靠近无关段落） | 调分块、Query 改写、混合检索 |
| **幻觉** | 检索不足时模型仍编造 | 拒答策略、强约束 Prompt、相似度防护栏 |
| **进阶召回** | 单靠向量不够稳 | **多路召回 / Hybrid**、**Rerank**、**Query 改写** |

### 2.4 动态防护栏（Guardrails）

在生成阶段约束答案须与检索材料一致；与“何时该用工具、何时拒答”的判断一脉相承，是 **防幻觉** 的工程化延伸。

## 三、架构演进：从「点状知识」到 LangChain 集成

### 3.1 实践背景

- **目标**：基于公司制度 **PDF** 集合做可溯源的智能问答。  
- **技术栈**：Python、**LangChain**、**FAISS**、阿里 **DashScope**（Embedding / 对话）、（笔记中亦提到 DeepSeek 等通过同一 API 体系调用）。

### 3.2 两阶段认知

**阶段一：点状知识（手动挡）**

- 关注如何把一段话变成向量：**Embedding**、相似度（如余弦）。  
- **痛点**：手写管理 `.model`/Pickle，向量与原文易错位，多文件难维护。

**阶段二：结构化集成（自动挡）**

- **FAISS（.faiss）**：向量检索 → **ID**。  
- **Docstore（.pkl）**：**ID → Document(page_content, metadata)**。  
- **Metadata**：**溯源**核心——`source_file`、`page_number`、`file_hash` 等。

### 3.3 踩坑与对策（摘要）

1. **语义稀释与 Chunk Size**：分片过大（如 3500）时背景噪声淹没主题 → 缩小 **chunk_size**，配合 **overlap** 防断层。  
2. **API 额度与重复 Embedding**：制度相对静态，全量重算浪费额度 → **MD5 文件指纹** + **processed 记录**，仅对新增或变更文件嵌入。  
3. **增量更新策略**：在已有索引上 **按源文件删除旧向量** 再 **add_texts**，与全量重建分工使用；需与 **元数据 `source_file`** 对齐，并处理 **源目录已删除文件** 的向量清理。

### 3.4 元数据与增量思路（代码级理解）

分块不仅存正文，还为每块打上 **文件级哈希** 等元数据，供增量比对；**sync** 流程在工程上体现为：比对哈希 → 仅对变化文件删旧加新 → 持久化 `processed_files.json` 与 FAISS。（具体实现以仓库 `langchain_rag.py` 为准。）

## 四、企业级制度 RAG：设计与实现要点

### 4.1 项目概述

基于 **LangChain** 的 RAG 原型：面向 **PDF 制度库** 的检索与问答；集成 **DashScope 向量化与对话模型**，强调 **增量更新** 与 **出处溯源**。

### 4.2 技术架构与选型（文档口径）

- **向量化**：阿里 embedding（文档示例为 **text-embedding-v4 / 高维** 等表述；**以当前 `langchain_rag.py` 中 `EMBEDDING_MODEL` 为准**）。  
- **大模型**：**qwen-turbo** / **deepseek-v3** 等（经 DashScope 调用；亦可 `DASHSCOPE_CHAT_MODEL` 配置）。  
- **向量库**：**FAISS** 本地落盘（`index.faiss` + `index.pkl`）。  
- **PDF**：**PyPDF2**，按页抽取以支持 **页码进 metadata**。

### 4.3 数据清洗与分块（ETL）

- **RecursiveCharacterTextSplitter**：**Chunk Size 500**、**Overlap 50**（与第三节笔记一致）。  
- **清洗**：去除不可见控制字符（如 `\x00-\x1f` 等区间）、规范化多余换行与 PDF 私用区符号，减轻 Token 噪声、利于 Embedding。

### 4.4 智能增量同步

- **`processed_files.json`**：`文件名 → MD5`，每次运行与磁盘 PDF 比对。  
- **变更**：对对应 `source_file` **删除旧 doc** 再 **嵌入新分块**；**源文件已删** 时按索引中的 `source_file` **清理孤儿向量**。  
- **冷启动 / 无 processed 与索引不一致**：按脚本策略 **全量重建** 或迁移，避免元数据与向量脱节。

### 4.5 相似度度量与过滤

- FAISS **IndexFlatL2** 返回距离；展示用 **单调映射**：  
  $$Similarity = 1 / (1 + L2\_Distance)$$  
  （避免未单位化向量下 LangChain 默认 `1 - d/√2` 出现负分误导。）  
- **`_DEFAULT_SIMILARITY_THRESHOLD`**：文档笔记中为 **0.4** 的表述；**以代码当前默认值为准**（用于过滤弱相关片段、降低胡编风险）。

### 4.6 问答策略（Prompt）

结构化 **System** 侧约束示例：

```plaintext
# ROLE: 严谨的公司行政助手
# CONSTRAINTS:
1. 仅根据【已知制度】回答，不得加入主观臆断。
2. 无法回答时，必须直说"库中暂无相关规定"。
3. 必须在回答末尾标注参考的文件名与页码。
```

### 4.7 命令行（`langchain_rag.py`）

| 命令 | 功能 | 典型场景 |
|------|------|----------|
| `python langchain_rag.py` | **交互式 RAG 问答** | 终端持续提问 |
| `python langchain_rag.py sync` | 同步 PDF → FAISS | 制度库更新后 |
| `python langchain_rag.py search [query]` | 只检索、打印片段 | 调检索与分数 |
| `python langchain_rag.py ask [query]` | 单次问答 | 脚本/集成 |
| `python langchain_rag.py help` | 帮助与示例 | 查用法 |

### 4.8 开发者避坑（Troubleshooting）

- **macOS OpenMP**：`KMP_DUPLICATE_LIB_OK=TRUE`，缓解 **FAISS + NumPy 等重复链接 libomp** 导致的崩溃。  
- **网络 / SSL**：DashScope **Embedding** 与 **Generation** 侧均可做 **有限次指数退避重试**（如 `dashscope_generation.py`）。  
- **Pickle 加载**：`FAISS.load_local(..., allow_dangerous_deserialization=True)` 需知悉来源可信。

### 4.9 小结（工程观）

- **数据质量**决定 RAG 上限；PDF 乱码再贵的 Embedding 也难救。  
- **工程**（增量、元数据、成本、拒答）往往比单纯换更大模型更关键。  
- **FAISS** 可视为「向量–ID」引擎，**灵魂在 metadata 与同步策略**。

🚀 RAG 从“入门”到“工业级”处理方案对比

| 阶段 | 方案名称 | 核心操作 | 解决的问题 | 成本/复杂度 | 稳定性 |
| --- | --- | --- | --- | --- | --- |
| 1. 入门级 | 原生 RAG | 直接切片 (Chunking) 后存入向量库 | 跑通基本流程 | 低 | ⭐ (极易产生幻觉) |
| 2. 进阶级 | 参数调优 | 调大 Top-K，优化 Chunk Size 和 Overlap | 减少上下文截断丢失 | 低 | ⭐⭐ |
| 3. 专业级 | 数据增强 | (目前所在阶段)在同步时强行拼接文件名、页码、章节标题 | 解决检索漂移、主语丢失 | 中 | ⭐⭐⭐⭐ (非常稳) |
| 4. 顶级 | 精排模型 (Rerank) | 向量检索 20 条 → 专用 Rerank 模型精排 Top-5 | 解决向量模型“词对词”匹配不精准的问题 | 中 (需部署 Rerank 模型) | ⭐⭐⭐⭐⭐ |
| 5. 高级 | 查询改写 (Pre-LLM) | 引入前置 LLM 将用户白话转为标准检索词 | 解决口语化、多轮对话指代不明 | 高 (多一倍 API 费) | ⭐⭐⭐⭐⭐ |
| 6. 终极 (最贵) | 混合检索 + 长上下文 | 向量检索 + 全文关键词检索 (BM25) + 极长上下文模型 | 解决所有长文本、复杂逻辑、表格解析问题 | 极高 (Token 消耗大) | 🏆 (天花板) |
