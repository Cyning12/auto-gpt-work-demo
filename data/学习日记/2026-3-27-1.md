📘 RAG 技术与应用：从理论到工业级实践的演进笔记
1. 实践背景
目标：构建一个基于公司规章制度 PDF 集合的智能问答助手。

技术栈：Python + LangChain + FAISS + 阿里 DashScope (Embedding) + DeepSeek (LLM)。

2. 核心知识点与架构演进
第一阶段：点状知识（手动挡）
最初我们关注的是如何把一段话变成向量。

Embedding：语义的数字表示。

相似度检索：通过余弦相似度找到最接近的文本。

痛点：手动管理 .model (Pickle) 文件，向量和原文的对应关系极易错位，难以处理多文件。

第二阶段：结构化集成（自动挡）
引入 LangChain 后，核心概念转向了 Document 对象。

FAISS 索引 (.faiss)：只负责查找 ID。

Docstore (.pkl)：负责 ID 到 Document(page_content, metadata) 的映射。

元数据 (Metadata)：解决“溯源”问题的关键，包含了文件名、页码、哈希值等。

3. 踩坑历程与深度思考
坑位 1：语义稀释与分片大小 (Chunk Size)
现象：最初设置 3500 字分片，搜索“桃园结义”搜到了“诸葛瞻成亲”。

思考：分片太大，背景噪音淹没了核心语义。

解决：将 chunk_size 调整为 500，并在 chunk_overlap 设置 50 个字符防止语义断层。

坑位 2：API 额度与重复计算的成本
现象：遇到 AllocationQuota.FreeTierOnly 报错，且每次运行脚本都要重新 Embedding 全量数据。

思考：公司制度是相对静态的，重复计算已存在的文件是对 API 额度的巨大浪费。

解决：引入 MD5 哈希校验 机制。

坑位 3：向量库的“增量更新”难题
现象：新增一个 PDF 后，不知道是该重构全量索引，还是只加一个补丁。

解决：利用 FAISS.load_local 加载旧索引 -> add_texts 加入新切片 -> save_local 覆盖。

B. 增量同步逻辑：避开“烧钱”陷阱
这是你 langchain_rag.py 中最精华的部分：先比对哈希，再决定是否调 API。
``` python
def sync_vector_store(source_folder, save_dir):
    # 1. 提取库中已有的文件哈希
    if vector_bundle_exists(save_dir):
        vector_db = FAISS.load_local(save_dir, embeddings, allow_dangerous_deserialization=True)
        processed_hashes = {doc.metadata.get("file_hash") for doc in vector_db.docstore._dict.values()}
    else:
        processed_hashes = set()

    # 2. 差集计算：只处理库里没有的哈希
    new_chunks = []
    for pdf_path in Path(source_folder).glob("*.pdf"):
        f_hash = get_file_hash(pdf_path)
        if f_hash not in processed_hashes:
            print(f"检测到新文件或更新：{pdf_path.name}")
            new_chunks.extend(extract_and_split_pdf(pdf_path, f_hash))
            
    # 3. 补丁式更新
    if new_chunks:
        vector_db.add_documents(new_chunks)
        vector_db.save_local(save_dir)

```

5. 总结与思考
数据的质量决定 RAG 的上限：
PDF 解析（尤其是表格）依然是痛点。如果 PdfReader 解析出来的东西乱码，后续 Embedding 再贵也救不回来。

工程重于模型：
对于企业应用，如何实现增量更新、如何做权限隔离（元数据过滤）、如何节省 Token 成本，这些工程设计比追求大模型参数更重要。

从“黑盒”到“灰盒”：
以前觉得 FAISS 是黑盒，现在明白它只是个“向量-ID”映射表，真正的灵魂在 .pkl 的元数据管理里。
