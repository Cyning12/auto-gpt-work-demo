前端 Code Review 指南（内部 mock 文档）
版本：v1.1 | 适用：Web / 小程序 H5
最后更新：2026-03-01

================================================================================
一、类型与接口
================================================================================
1. 所有对外 props、API 响应 DTO、Store 状态必须提供 TypeScript 接口或 type，禁止 any 兜底（除非有 eslint-disable 与注释说明原因）。
2. 组件 props 变更需同步更新 Storybook / 文档站中的示例。
3. 枚举优先使用 const enum 或字面量联合类型，避免魔法字符串散落。

================================================================================
二、React / 状态
================================================================================
1. 禁止在循环、条件分支内直接调用 setState 或修改可变引用型 State（应使用不可变更新或 reducer）。
2. 超过 300 行的单文件组件必须拆分：UI 容器、业务 Hook、纯展示子组件至少拆成 2 个文件。
3. useEffect 依赖数组必须完整；禁止空依赖却读取外部可变值的「陈旧闭包」写法。

================================================================================
三、性能与体验
================================================================================
1. 列表页超过 50 条需虚拟列表或分页；图片需指定 width/height 或 aspect-ratio，避免 CLS 抖动。
2. 首屏关键路径禁止同步引入重型库；路由级 code splitting 为默认要求。
3. 表单提交需有 loading / 禁用态，防止重复提交。

================================================================================
四、测试与质量
================================================================================
1. 核心业务逻辑（计费、权限、订单状态机）需 Jest 单元测试，覆盖率门禁不低于 70%（关键模块）。
2. E2E 仅覆盖主路径冒烟；复杂场景以单测 + 接口契约为准。
3. PR 必须通过 ESLint、Prettier、TypeScript --noEmit。

================================================================================
五、安全与可访问性
================================================================================
1. 用户输入展示前需转义或白名单渲染，禁止 dangerouslySetInnerHTML 无消毒。
2. 可点击区域最小 44×44px；表单控件需有 label 或 aria-label。
3. 敏感操作（删除、支付）需二次确认或撤销窗口。

================================================================================
六、Review 检查清单（Reviewer 勾选）
================================================================================
[ ] 类型完整，无隐式 any
[ ] 无循环内改 State
[ ] 大组件已拆分或已登记技术债 Issue
[ ] 关键逻辑有单测或已说明豁免原因
[ ] 无已知 XSS / a11y 红线问题
