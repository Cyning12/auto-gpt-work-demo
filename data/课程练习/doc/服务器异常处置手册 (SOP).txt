服务器与数据库异常处置手册（SOP）（内部 mock）
文档编号：SOP-IT-INFRA-2026
适用范围：生产环境 MySQL / 应用服务器 / 负载均衡

================================================================================
一、总则
================================================================================
1. 值班工程师先确认告警来源（监控系统 / 用户反馈 / 日志平台），记录告警时间与影响范围。
2. 所有变更操作须在值班群同步；涉及数据删除或主从切换需二线负责人审批。
3. 紧急联系人（mock）：DBA 张三 138-0013-8000；基础架构李四 139-0013-9000。

================================================================================
二、场景 A：数据库连接数暴涨（原 SOP-IT-01）
================================================================================
现象：连接数逼近或达到 max_connections，应用报 “Too many connections”。
步骤：
1. 登录主库，执行 SHOW VARIABLES LIKE 'max_connections'; 与 SHOW STATUS LIKE 'Threads_connected';
2. 执行 SHOW FULL PROCESSLIST，按 Time 降序；对明显异常、长时间 Sleep 且来自错误应用的连接，经确认后可 KILL；
3. 若慢查询导致连接堆积，临时开启慢查询日志或从 Performance Schema 定位 SQL，联系业务方限流或加索引；
4. 仍无效或疑似连接泄漏：通知 DBA 张三，评估短时调大 max_connections（需变更单）与发布回滚。

================================================================================
三、场景 B：CPU 持续高于 85%
================================================================================
步骤：
1. top / pidstat 确认是 mysqld、应用进程还是批处理任务；
2. 应用进程：检查是否发布新版本、线程 dump、GC 日志（若为 JVM）；
3. 数据库：结合慢查询与 buffer pool 命中率；必要时只读实例承担报表查询；
4. 超过 30 分钟无缓解：启动扩容或限流预案，并登记事故单。

================================================================================
四、场景 C：内存使用率告警（>90% 持续 5 分钟）
================================================================================
步骤：
1. 区分是缓存正常占用还是泄漏（观察趋势是否单调上升）；
2. 应用侧：检查大对象缓存、会话堆积；必要时滚动重启（需避开高峰）；
3. 数据库：检查 tmp_table_size、排序内存；避免大事务长连接占满内存。

================================================================================
五、场景 D：磁盘空间不足（/data 或 binlog 分区）
================================================================================
步骤：
1. df -h 与 du 定位目录；清理过期日志（按保留策略）、压缩归档；
2. binlog：在确认已从库同步后，按运维规范 purge；
3. 禁止手工 rm 正在写入的数据库数据文件；扩容或迁移走变更流程。

================================================================================
六、场景 E：主从延迟过大
================================================================================
步骤：
1. SHOW SLAVE STATUS\G 查看 Seconds_Behind_Master、Slave_IO_Running / Slave_SQL_Running；
2. 单表大事务：协调业务拆分事务；从库只读压力过大时考虑读写分离权重调整；
3. 硬件或网络问题：切换链路或提升从库规格（变更窗口执行）。

================================================================================
七、事后要求
================================================================================
1. P1/P2 事故 24 小时内提交简要复盘（根因、时间线、改进项）。
2. 涉及配置变更需更新 CMDB 与 Runbook 对应条目。
